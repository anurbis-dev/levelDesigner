## Главный факт, который определяет весь план

Редактор уже честно зафиксировал границу ответственности в `ASSET_TYPES_CATALOG.md`:
> "Runtime-поведение (коллизии, AI, диалоги и т.д.) не реализуется редактором — это ответственность
> игры, потребляющей экспортированный JSON уровня."

То есть данные уже есть (28 типов ассетов + 19 типов компонентов, сериализуются в `objects[].type`,
`objects[].components[]`, `asset.components[]`), а вся runtime-логика — 0% реализована. Движок не
проектируется "с нуля в вакууме" — он проектируется как **интерпретатор уже существующей схемы**.
Это меняет порядок работы: сначала зафиксировать контракт схемы, потом писать код, который её читает.

### 0.1 Документ `docs/RUNTIME_SCHEMA.md`
Для каждого из 28 asset-типов и 19 component-типов зафиксировать:
- какие поля в `properties`/`components[].properties` ожидает движок (сейчас это `{}` — пустой стаб);
- обязательные vs опциональные поля, дефолты;
- версию схемы каждого типа отдельно (не только версию уровня целиком) — типы будут расширяться
  независимо друг от друга, и у движка должна быть возможность понять "это старый `collider` без
  поля `shape`" и не упасть.

Это не работа на неделю по всем 47 типам сразу — заполняется по мере реализации в Фазе 3, но
**каркас документа и процесс его ведения** нужно завести сейчас, иначе спецификация будет жить
только в головах и в код-ревью, как сейчас живёт факт "components — пока просто стаб".

### 0.2 Версionирование формата
`Level.meta.version` уже существует (`data.meta?.version || 'dynamic'`) — сейчас не используется
осмысленно (дефолт `'dynamic'` ничего не версионирует). Заменить на реальный semver/int формата
уровня. Loader движка (Фаза 2) обязан проверять версию и явно фейлиться с понятной ошибкой на
несовместимом формате, а не молча падать на `undefined.foo`.

### 0.3 Модель контента: Project — реальная runtime-сущность, не editor-only

**Пересмотрено.** Движок грузит не отдельный `level.json`, а `Project` целиком — Project это игра
(ассеты + уровни + логика внутри уровней + настройки проекта), Engine — универсальный рантайм,
ничего не знающий про конкретную игру. Связка `Engine + Project = запущенная игра`, аналогично
паре "движок-бинарник + `project.godot`" у Godot или Unity Engine + `Assets/`.

**Ключевая развилка, которую нельзя пропустить:** editor-`Project` (набор открытых вкладок-уровней
в редакторе, уже реализовано в multi-level плане) и runtime-`Project` (то, что грузит движок) —
**не один и тот же объект**, даже если названы одинаково:

| | editor-Project | runtime-Project |
|---|---|---|
| Что содержит | все уровни, открытые в табах прямо сейчас (черновики, тестовые сцены — тоже) | только то, что явно помечено как часть релиза |
| Кто пишет | `ProjectFileOperations` (уже реализовано) | отдельный экспорт-шаг (Фаза 4) |
| `viewState`, `currentLevelIndex` | нужны (какая вкладка активна в UI) | не нужны в игре вообще |

Если движок будет напрямую грузить editor-`Project.toJSON()`, черновики из открытых вкладок
утекут в игру, а editor-only поля (viewState) будут мусором в рантайме. Нужен явный **export step**:
`ProjectExporter.export(editorProject) → runtimeProjectManifest`, который берёт только уровни,
явно помеченные разработчиком как "включено в билд" — то же понятие, что уже упоминалось для
`levels-manifest.json` в Фазе 4, теперь формализованное на уровне Project, а не разрозненным списком.

### 0.4 Аддоны и спец-события — три типа контент-паков

По итогам уточнения: у Project два разных механизма расширения с разной семантикой, плюс сам факт
обновления игры — это не расширение, а версионирование:

**Патч базового Project** — не отдельная сущность, а новая версия самого Project (см. 0.2,
versioning формата). Обновление игры = замена содержимого runtime-Project более новой версией
целиком (или дельта-патч конкретных файлов) — не проходит через Addon-механизм ниже.

**Addon** — переопределяющее расширение:
```
{
  id: "addon-winter-skins",
  version: "1.0.0",
  targetProjectId: "base-game",
  targetProjectVersion: "^1.4.0",   // проверка совместимости при загрузке
  overrides: {
    assets: { "<assetId>": "path/to/replacement-asset.json" },
    levels: { "<levelId>": "path/to/replacement-level.json" }
  },
  additions: {
    assets: [ /* новые ассеты, которых нет в базе */ ],
    levels: [ /* новые уровни */ ]
  }
}
```
Переопределение — **по `id`, не по `path`**. Это критично: `Asset.id` уже стабилен и не зависит
от пути (`Asset.js` генерирует `id` отдельно от `path` — переименование/перенос ассета его не
меняет) — на нём можно строить override. А вот `Level.toJSON()` **сейчас не сериализует
собственный `id`** (это уже всплывало как проблема при реализации `Project` в multi-level плане —
`id` уровня пришлось протаскивать отдельно через `Project.toJSON()`, потому что `Level.toJSON()`
его не отдаёт). Для Addon-системы это больше не второстепенная деталь: **без стабильного,
известного движку id уровня переопределение уровня аддоном невозможно.** Два варианта:
добавить `id` в сам `Level.toJSON()`, либо гарантировать, что runtime-Project-манифест (0.3)
сам хранит пару `levelId → путь к файлу` отдельно от содержимого уровня — тогда `id` не обязателен
внутри `level.json`, достаточно, что его знает манифест Project. Второй вариант меньше меняет
существующий формат — предпочтителен.

Порядок применения при нескольких Addon: явный список с приоритетом (не порядок файлов на диске),
конфликт двух аддонов на один `id` — побеждает выше приоритет, с warning в лог загрузки, не
silent overwrite.

**Special Event** — параллельный, изолированный контент-пак:
```
{
  id: "winter-event-2026",
  version: "1.0.0",
  requiresProjectId: "base-game",
  requiresProjectVersion: "^1.4.0",  // может читать ассеты базы, но не переопределяет их
  levels: [ /* собственные уровни */ ],
  rules: { /* специфичные для события правила/условия */ }
}
```
Ключевое отличие от Addon: **никаких `overrides`** — Special Event не трогает содержимое базового
Project, только добавляет отдельный, самостоятельный набор уровней+правил, живущий параллельно
основной игре (отдельная точка входа в меню, вероятно свой namespace сохранений, чтобы не мешать
прогрессу основной кампании). Патч на основную игру — отдельный процесс (новая версия Project),
не связан с выпуском/снятием Special Event.

**Критерий готовности Фазы 0:** есть `docs/RUNTIME_SCHEMA.md` (0.1), `meta.version` осмысленно
проверяется (0.2), есть явный `ProjectExporter` (editor-Project → runtime-Project-манифест, 0.3),
три типа паков (Project/Addon/Event) с их manifest-схемами зафиксированы в `docs/CONTENT_MODEL.md`
(0.4), включая решение по хранению `levelId` в манифесте Project.

21 vitest-тест (70→91). Три решения по ходу реализации:
- **`src/engine/` полностью самодостаточен** — ноль импортов из `src/constants|core|
  managers|models|ui|utils|widgets` (весь текущий `src/` и есть "editor" в терминах границы

  4 компонента), сетевая загрузка манифеста (`fetch` — Фаза 3/4; `loadProject(manifest)`

Ниже — исходные §1.0-1.3 плана как есть (реализация им соответствует 1:1, см. выше).

Ровно блоки 1–8 из общей архитектуры движка (Scene/Entity, Level Loader, Game Loop, Renderer,
Asset Loader, Input, Component-система, Behavior Registry) — без изменений по сути, здесь

### 1.0 `ProjectLoader` — точка входа движка (заменяет прямую загрузку `level.json`)
Порядок загрузки при старте `GameEngine`:
1. Загрузить манифест базового `Project` (0.3), построить реестр `assetsById`, `levelsById`.
2. Применить список активных `Addon` в порядке приоритета (0.4) — для каждого `overrides.assets[id]`/
   `overrides.levels[id]` заменить запись в реестре, для `additions` — добавить новую. Конфликт по id
   между несколькими аддонами — лог warning, побеждает последний по приоритету.
3. Зарегистрировать доступные `Special Event` паки отдельно, **не сливая** их уровни/ассеты в
   основной реестр — держать в собственном `eventsById`, чтобы основная игра и события не могли
   случайно друг друга задеть по id.
4. Только после этого — грузить конкретный уровень (по умолчанию entry-level из Project) через
   `EntityFactory` (1.1) ниже. Для MVP (Фаза 1) достаточно Project с одним уровнем и без аддонов/
   событий — сама многослойность загрузки (шаги 1–3) нужна как каркас с первого дня, чтобы её не
   пришлось переделывать, когда появится первый реальный Addon, но **наполнять** Addon/Event-логику
   в MVP не нужно.

### 1.1 `EntityFactory` — мост `GameObject`/`Group` (editor model) → `Entity` (engine model)
Ключевая развилка форматов:
- `GameObject` (редактор): `{id, name, type, x, y, width, height, color, rotation, imgSrc,
  visible, locked, layerId, properties, components}` — плоская структура, `type` это ключ в
  каталог `AssetTypes.js` (`'actor'`, `'camera'`, `'tilemap'` и т.п.), `components[]` — массив
  `{id, type, enabled, properties}`.
- `Entity` (движок): та же структура минус editor-only поля (`locked` не нужен в рантайме,
  `visible` — нужен, но означает не "скрыт в редакторе", а активен ли рендер в игре).

`EntityFactory.fromGameObjectData(data)` — единая точка конвертации, не размазывать парсинг по
всему движку. Здесь же place для валидации против `docs/RUNTIME_SCHEMA.md`.

### 1.2 Renderer — держать паритет с editor rendering, а не переизобретать
Редактор уже решил нетривиальные вещи, которые движку тоже понадобятся один в один, иначе
"что видно в Play-in-editor" разойдётся с тем, что видно в собранной игре:
- **Layers**: рендер по `layers[]` с сортировкой, видимостью — та же модель, что в
  `RenderOperations.js`/`Layer.js`, только без editor-only флагов (lock/solo/isolate).
- **Parallax**: `Level.settings.parallaxHorizontal/parallaxVertical` + per-layer parallax уже
  сериализуются в уровень (см. changelog v3.57.0) — если движок это проигнорирует, Play-in-editor
  будет выглядеть иначе, чем сохранённый файл в редакторе. `ParallaxRenderer.getCameraOffset()`
  логику нужно портировать (не импортировать из editor/, а реализовать заново в engine/, по тем
  же формулам — граница `engine` не должна тянуть editor-код).
- **Camera asset** (`camera` type, `AssetTypes.js`): FOV/zoom, follow-target, границы, слои
  рендера — это ровно блок 11 (Camera) общей архитектуры, но конкретные поля уже определены
  каталогом, не нужно придумывать заново.
- **Новое прикладное prior art (Фаза B4.2 редактора, split-tree docking, ещё не закоммичено):**
  `src/core/ViewportViewManager.js` уже различает `CameraSource` `{kind:'work'}` (навигация
  редактора) vs `{kind:'game', objectId}` (поза берётся с `camera`-типа `GameObject` через
  `resolveCamera()`) — та же развилка work-camera/game-camera, что движку нужна для блока 11.
  Формулы самой камеры (`resolveCamera`) стоит свериться на совпадение с тем, что заложит движок,
  а не изобретать точки схождения заново в Фазе 3.

### 1.3 Asset Loader — типы, требующие реальной загрузки файлов
Из 28 asset-типов не все требуют engine-side загрузки контента:
- Требуют загрузки: `image`, `imageAtlas`, `soundEffect`, `musicTrack`, `fontTextStyle`,
  `tileset` (атлас + сетка), `spriteAnimationClip` (набор кадров).
- НЕ требуют загрузки файлов, только данные из JSON: `camera`, `volume`, `dialogueGraph`,
  `questObjective`, `itemDefinition`, `inventorySchema`, `pathSpline`, `aiBehaviorPreset` и т.д. —
  это структуры данных, не бинарные ассеты.

Asset Loader (блок 5) должен явно различать эти две категории, а не пытаться "загружать" всё
подряд по единой логике.

**Критерий готовности Фазы 1:** минимальный runtime-Project (один уровень, без объектов игровых
типов, просто `layers` и `settings`) грузится через `ProjectLoader` → `EntityFactory`, рендерится
с правильным фоном/параллаксом, камера двигается. Ещё без единого работающего геймплейного
поведения и без единого реального Addon/Event — только пройденный насквозь каркас загрузки.

## Фаза 2 — Behavior Registry + первые 4 компонента (вертикальный срез)

**Зачем именно вертикальный срез:** 19 component-типов и 28 asset-типов реализовывать все сразу —
гарантированный способ никогда не дойти до работающей игры. Правильный порядок — взять минимальный
набор, достаточный для одного реально играбельного уровня, и только потом расширять горизонтально.

### 2.1 Реестр
```
BehaviorRegistry.register('collider', ColliderBehavior);
BehaviorRegistry.register('playerStart', PlayerStartBehavior);
...
```
Резолвится по `component.type` при инстанцировании `Entity` из `EntityFactory` (1.1). Незнакомый
`type` (component ещё не реализован в движке) — не должен крашить загрузку уровня, только
warning в консоль ("component type 'aiBehaviorPreset' not implemented, skipped") — иначе
редактор и движок жёстко залочены на синхронную разработку каждого типа, что убивает саму идею
независимости.

### 2.2 Первые 4 компонента для MVP (рекомендация, исходя из каталога)
- `collider` — без него ничего не взаимодействует физически, это буквально блок 9 (Collision)
  общей архитектуры, реализуемый как компонент, а не отдельная система.
- `playerStart` — точка появления игрока, нужна для того, чтобы вообще был играбельный уровень.
- `trigger` — зона события (в паре с `volume` asset-типом, который уже задуман как "зона-триггер
  произвольной формы").
- `interactable` — базовое взаимодействие игрока с объектом (открыть/подобрать/активировать) —
  минимальный geймплейный глагол, от которого дальше расширяются `pickup`, `dialogueTrigger`,
  `movablePushable` и т.д. как частные случаи.

### 2.3 Остальные 15 компонентов — бэклог, не блокируют релиз
`spriteUiAnimation`, `transformAnimation`, `pickup`, `dialogueTrigger`, `damageHealth`,
`movablePushable`, `mountableVehicleSeat`, `pathFollower`, `spawner`, `stateMachineBehavior`,
`checkpointSavePoint`, `climbableLadder`, `conveyorZiplineJumpPadPortal`, `destructibleContainer`,
`variableModifier` — заносятся в `docs/RUNTIME_SCHEMA.md` (Фаза 0.1) со статусом "не реализовано",
реализуются по мере того, какие реально нужны конкретной игре, которую строите на этом движке —
не абстрактно "чтобы каталог был полным".

**Критерий готовности Фазы 2:** можно построить в редакторе уровень с игроком (playerStart),
стенами (collider) и одним триггером/интерактивным объектом, и он реально играбелен через Play-in-editor.

## Фаза 3 — Play-in-editor (замыкает контур editor ⇄ engine)

Механизм уже описан в прошлом обсуждении архитектуры (сериализация текущего `Level` в тот же JSON,
новый чистый инстанс движка в отдельном canvas/iframe). Здесь — что конкретно нужно с учётом
multi-level:

### 3.1 Что сериализовать при Play
Play собирает **временный runtime-Project "на лету"**, не полноценный экспорт (Фаза 4): один
текущий уровень (`editor.level`, через computed getter → `levelSessions.get(currentLevelId).level`)
как единственный уровень минимального Project-манифеста, без Addon/Event — иначе Play пришлось бы
каждый раз гонять полный `ProjectExporter` по всем открытым вкладкам, что и медленнее, и не нужно:
Play проверяет один уровень, над которым сейчас идёт работа, а не сборку игры целиком. Полная
сборка со всеми уровнями/аддонами — только явная команда "Build" (Фаза 4), не Play.

### 3.2 Кнопка Play
- Сериализация: `editor.level.toJSON()` — без изменений формата, тот же метод, что и "Save".
- Валидация перед запуском: есть ли хотя бы один объект типа `playerStart`. Если нет — не запускать
  движок вникуда, показать понятную ошибку в статус-баре (`Logger.status.warn`, как уже принято в
  проекте для похожих edge cases).
- Новый инстанс `GameEngine` в отдельном `<canvas>` поверх/вместо editor-канвы, старый editor-канвас
  не уничтожается, только скрывается (стоп — просто убить движок и показать канвас обратно).
- **Механизм уже есть в редакторе (Фаза B4.2, `CanvasRenderer.setTarget()`/`restorePrimaryTarget()`,
  `RenderOperations.render()` перебирает `viewportViewManager.getViews()`):** мульти-canvas рендер
  без потери primary-канваса реализован для editor-side multi-viewport. Play-in-editor может
  переиспользовать тот же паттерн (ещё один "view" с canvas, который рисует не редактор, а
  `GameEngine`) вместо отдельного оверлей-механизма поверх/вместо editor-канвы.

**Критерий готовности:** уровень, собранный в редакторе, запускается кнопкой Play без пересборки/
экспорта, ведёт себя идентично тому, что получится в финальной сборке (Фаза 4).

## Фаза 4 — Релизная сборка

Тоже уже согласовано архитектурно (esbuild, `player.html`, `eslint-plugin-boundaries` на
`engine/` не импортирует `editor/`) — здесь конкретика под текущий проект:

### 4.1 Bundle entry point
`src/engine/index.js` — экспортирует только `GameEngine`, `EntityFactory`, `BehaviorRegistry` и
публичный API загрузки уровня. Ничего из `src/editor/**`, `src/ui/**`, `src/managers/**`.

### 4.2 Данные для релиза — три отдельные сборочные цели
`ProjectExporter` (0.3) — единственная точка правды о том, что попадает в релиз, используется
всеми тремя:

- **`npm run build:game`** — базовый Project: `ProjectExporter.export(editorProject)` строит
  `project.json` (манифест с `levelId → путь`, см. 0.4) + только уровни, явно помеченные
  разработчиком как "включено в билд" (**не** автоматически всё из `content/maps`, чтобы
  черновики/тестовые уровни не утекали в релиз) + только ассеты, реально используемые этими
  уровнями. Отсечение мёртвого контента — asset-usage-граф ("уровень → какие ассеты использует"),
  тот же принцип, что уже применялся для дедупликации кода (madge/jscpd), только на данных, а не
  на импортах.
- **`npm run build:addon -- --id=<addonId>`** — то же самое, но выход `addon.json` с `overrides`/
  `additions` (0.4) вместо полного Project-манифеста; требует, чтобы разработчик явно указал в
  редакторе, какие ассеты/уровни этого workspace — переопределения существующих (по id), а какие —
  новый контент.
- **`npm run build:event -- --id=<eventId>`** — аналогично, выход `event.json`, без `overrides`
  вообще (0.4) — только `levels`/`rules`.

Все три используют один и тот же собранный `engine/index.js` (4.1) — движок один, паки разные.

**Критерий готовности Фазы 4:** `npm run build:game` производит папку `dist/game/`, которая
открывается локально (`file://` или простой static server) без единого файла/импорта из
`editor/`, содержит `project.json` и только контент из явного манифеста. `build:addon`/
`build:event` пока могут быть заглушками (команда существует, реализуется когда появится первый
реальный аддон/событие) — важно, что формат вывода уже согласован с 0.4, а не изобретается заново
в момент первого реального аддона.

## Сводка порядка работ

| Фаза | Что | Блокирует |
|---|---|---|
| 0 | Контракт схемы + версия формата + модель Project/Addon/Event + ProjectExporter | Всё остальное |
| 1 | MVP-ядро (ProjectLoader/Scene/Loop/Renderer/AssetLoader/Input) без геймплея | Фазу 2 |
| 2 | Behavior Registry + 4 компонента вертикальным срезом | Фазу 3 |
| 3 | Play-in-editor (Project с одним текущим уровнем) | Фазу 4 (проверка WYSIWYG) |
| 4 | Релизная сборка: build:game / build:addon / build:event | — |

Остальные 15 компонентов и ~20 нереализованных asset-типов (tileset, particleEffect, dialogueGraph,
navMesh и т.д.) — не отдельная фаза с дедлайном, а бэклог в `docs/RUNTIME_SCHEMA.md`, который
закрывается по одному типу за коммит, по мере того как конкретная игра, которую вы делаете поверх
этого движка, реально в них упирается — та же дисциплина, что применялась к God Object'ам и
дублированию кода в плане рефакторинга: не строить то, что не нужно прямо сейчас.
